## 🚨 避坑记录

### 坑 1: 路径差异警报

**问题**: 公司端与家里端 .plist 文件名不一致

| 端点 | 文件名 |
|------|--------|
| 家里端 | `com.user.agfs.plist` |
| 公司端 | `com.user.agfs.server.plist` |

**影响**: 运维脚本硬编码文件名会导致一端成功、一端失败

**解决方案**:
```bash
# ❌ 错误：硬编码文件名
launchctl load ~/Library/LaunchAgents/com.user.agfs.plist

# ✅ 正确：先检测文件名存在性
PLIST_FILE=""
if [ -f "$HOME/Library/LaunchAgents/com.user.agfs.plist" ]; then
    PLIST_FILE="$HOME/Library/LaunchAgents/com.user.agfs.plist"
elif [ -f "$HOME/Library/LaunchAgents/com.user.agfs.server.plist" ]; then
    PLIST_FILE="$HOME/Library/LaunchAgents/com.user.agfs.server.plist"
fi

if [ -n "$PLIST_FILE" ]; then
    launchctl load "$PLIST_FILE"
else
    echo "❌ 未找到 LaunchAgent 配置文件"
    exit 1
fi
```

**Gene 固化**: `gene_global_003_path_detection`

### 坑 2: 端口冲突逻辑

**问题**: launchctl 守护下，手动启动 Server 会导致 `address already in use`

**场景**:
```bash
# LaunchAgent 已在运行
launchctl list | grep agfs
# 输出: com.user.agfs

# 错误：手动启动
~/.local/bin/agfs-server -c config.yaml
# 结果: bind: address already in use
```

**根本原因**: LaunchAgent 和手动启动争夺 1933 端口

**解决方案**:
```bash
# ❌ 错误：手动启动（会导致冲突）
nohup ~/.local/bin/agfs-server -c config.yaml &

# ✅ 正确：通过 launchctl 管理
# 重启
launchctl kickstart -k gui/$(id - u)/com.user.agfs

# 或先卸载再加载
launchctl unload ~/Library/LaunchAgents/com.user.agfs.plist
launchctl load ~/Library/LaunchAgents/com.user.agfs.plist
```

**自愈脚本规范**:
```bash
#!/bin/bash
# AGFS 自愈脚本 - 规范版

restart_agfs() {
    # 1. 检测 LaunchAgent 是否存在
    if launchctl list | grep -q "com.user.agfs"; then
        # 使用 launchctl 重启
        launchctl kickstart -k gui/$(id - u)/com.user.agfs
        echo "✅ 已通过 launchctl 重启"
    else
        # 手动启动（仅作为兜底）
        ~/.local/bin/agfs-server -c ~/.openviking/AGFS_GOLDEN_V1.yaml \
            > ~/Documents/openclaw/logs/agfs.server.log 2>&1 &
        echo "✅ 已手动启动"
    fi
}
```

**Gene 固化**: `gene_global_004_launchctl_management`

### 坑 3: 重定向陷阱

**问题**: `>` 重定向无法自动创建不存在的目录

**错误场景**:
```bash
# 目录不存在
~/.local/bin/agfs-server > ~/Documents/openclaw/logs/agfs.server.log 2>&1
# 结果: No such file or directory
```

**根本原因**: `>` 只会创建文件，不会创建父目录

**解决方案**:
```bash
# ❌ 错误：直接重定向
~/.local/bin/agfs-server > ~/Documents/openclaw/logs/agfs.server.log 2>&1

# ✅ 正确：先创建目录
mkdir -p ~/Documents/openclaw/logs
~/.local/bin/agfs-server > ~/Documents/openclaw/logs/agfs.server.log 2>&1

# ✅ 更好：封装函数
ensure_log_dir() {
    LOG_DIR="$HOME/Documents/openclaw/logs"
    if [ ! -d "$LOG_DIR" ]; then
        mkdir -p "$LOG_DIR"
        echo "✅ 创建日志目录: $LOG_DIR"
    fi
}

start_agfs() {
    ensure_log_dir
    ~/.local/bin/agfs-server -c ~/.openviking/AGFS_GOLDEN_V1.yaml \
        > ~/Documents/openclaw/logs/agfs.server.log 2>&1 &
}
```

**Gene 固化**: `gene_global_005_directory_creation`

## 🧬 进化引擎更新

### 新增 Gene

```yaml
# gene_global_003_path_detection
description: 路径存在性检测
rule: 所有运维脚本必须先检测文件/目录存在性，禁止硬编码
triggers:
  - 编写脚本时
  - 部署到多端时
actions:
  - 使用条件判断检测路径
  - 提供 fallback 方案
  - 失败时给出明确错误信息

# gene_global_004_launchctl_management
description: LaunchAgent 管理规范
rule: 优先使用 launchctl 管理守护进程，禁止手动启动冲突
triggers:
  - 重启 AGFS Server 时
  - 自愈脚本执行时
actions:
  - 检测 LaunchAgent 是否存在
  - 使用 kickstart 或 unload/load
  - 手动启动仅作为兜底

# gene_global_005_directory_creation
description: 目录自动创建
rule: 所有涉及文件/日志的操作必须先执行 mkdir -p
triggers:
  - 创建日志文件时
  - 创建配置文件时
  - 启动服务时
actions:
  - 先创建父目录
  - 验证目录权限
  - 失败时给出明确错误信息
```
