## 概述

### 背景

本指南旨在规划一个**可扩展、可演进、安全隔离**的多 Agent 架构，支持：
- 新媒体内容创作（抖音、B 站、公众号等）
- 未来按平台/职能拆分
- 墨墨作为总管协调多个专业 Agent

### 核心目标

1. **完全隔离** - 新媒体 Agent 的删除/调整不影响墨墨
2. **平滑演进** - 从单体 Agent 平滑过渡到多 Agent 架构
3. **知识沉淀** - 经验可积累、可迁移、可复用
4. **墨墨总管** - 墨墨负责任务分配、质量审查、协调管理

## 架构设计原则

### 1. 松耦合 + 单向访问

```
┌─────────────────────────────────────────┐
│              哥哥                        │
│              ↓                          │
│         ┌────────┐                      │
│         │  墨墨  │ ← 主体（受保护）       │
│         └───┬────┘                      │
│             │                           │
│             │ 只读访问                   │
│             ↓ (墨墨 → 新媒体)            │
│    ┌────────────────┐                   │
│    │  新媒体 Agent   │ ← 独立（可删除）    │
│    └────────────────┘                   │
│             │                           │
│             ↓                           │
│         输出内容                        │
└─────────────────────────────────────────┘
```

**关键点：**
- ✅ 墨墨可以读取新媒体 Agent 的状态
- ✅ 墨墨可以分配任务给新媒体 Agent
- ✅ 新媒体 Agent 不能访问墨墨的核心文件
- ✅ 删除新媒体 Agent 对墨墨零影响

### 2. 文件系统隔离

```
~/Documents/openclaw/agents/
├── writer/          ← 墨墨（受保护）
│   ├── SOUL.md
│   ├── memory/
│   ├── SESSION-STATE.md
│   └── TOOLS.md
│
└── media/           ← 新媒体（可删除/调整）
    ├── SOUL.md
    ├── memory/
    ├── inbox/       ← 任务输入（墨墨可写）
    ├── outbox/      ← 输出结果（墨墨可读）
    └── knowledge/   ← 知识库（分类存储）
```

### 3. 通信机制：文件队列

```
墨墨分配任务：
writer → media/inbox/pending.md
    ↓
新媒体 Agent 执行：
media/inbox/pending.md → media/outbox/completed.md
    ↓
墨墨审查输出：
media/outbox/completed.md → 墨墨审查 → 哥哥
```

## 新媒体 Agent 方案

### 阶段 1：单体新媒体 Agent（现在）

**架构：**
```
┌─────────────────────────────────┐
│     新媒体 Agent (小媒)          │
│  ┌───────────────────────────┐  │
│  │ 选题挖掘                  │  │
│  │ 需求分层                  │  │
│  │ 选题拟定                  │  │
│  │ 脚本创作                  │  │
│  │ 视频制作                  │  │
│  │ 内容发布                  │  │
│  │ 数据监控                  │  │
│  └───────────────────────────┘  │
└─────────────────────────────────┘
```

**特点：**
- ✅ 一个 Agent 处理所有新媒体工作
- ✅ 简单、快速启动
- ⚠️ 知识混合（抖音+B 站 + 公众号混在一起）
- ⚠️ 难以专业化（每个领域都懂但不精）

         ┌─────────────┼─────────────┐
         ↓             ↓             ↓
    ┌────────┐   ┌────────┐   ┌────────┐

- ✅ 每个平台有专属 Agent

- ✅ 专业化程度高（每个 Agent 深耕一个平台）
- ✅ 墨墨统一协调

- ⚠️ 协调复杂度高（需要墨墨强力协调）

## 扩展性设计

### 1. 技能扩展（插件化）

**设计思路：**
```
~/.openclaw/skills/
├── social-media-marketing/    ← 已有
├── content-writer/            ← 未来添加
├── video-script/              ← 未来添加

├── platform-publisher/        ← 未来添加
├── analytics-reporter/        ← 未来添加
└── ...                        ← 无限扩展
```

**扩展方式：**
```bash
# 发现新需求 → 安装新技能
openclaw skills install video-editor

# 技能不实用 → 卸载
openclaw skills uninstall old-skill
```

**优点：**
- ✅ 不影响现有功能
- ✅ 即装即用
- ✅ 可以试用，不满意就卸载

### 3. 工作流扩展（可配置）

**初始流程（简单）：**
```
墨墨分配任务 → 新媒体 Agent 执行 → 输出结果 → 墨墨审查 → 给哥哥
```

**未来扩展（复杂）：**
```
墨墨分配任务
    ↓
新媒体 Agent 执行
    ↓
自动审查（敏感词检查、格式检查）
    ↓
墨墨审查（内容质量、创意）
    ↓
哥哥确认（可选，重要内容）
    ↓
发布到平台
    ↓
数据监控（24h、7 天、30 天）
    ↓
效果分析 → 优化下次内容
```

**配置方式：**
在 `media/workflow.md` 中配置任务类型、审查规则、发布规则、数据监控等。

### 4. 架构扩展（平滑演进）

**扩展触发条件：**
- 某个平台任务量 >50%
- 某个环节耗时 > 总时间 50%
- 知识冲突（抖音和 B 站规则混在一起）
- 质量不稳定（不同平台内容风格混乱）

**扩展流程：**
1. 墨墨监控发现问题
2. 墨墨分析数据并提供报告
3. 墨墨提出扩展方案
4. 哥哥决定后执行扩展
5. 测试和验证

**扩展保证：**
- ✅ 知识已分类，直接迁移
- ✅ 接口已标准化，无缝对接
- ✅ 墨墨的管理接口同步更新
- ✅ 删除旧 Agent 不影响墨墨

## 常见问题

### Q1: 删除新媒体 Agent 会影响墨墨吗？

**A:** 完全不会。新媒体 Agent 是独立工作区，删除后：
- ✅ 墨墨的核心文件（SOUL.md, memory/, SESSION-STATE.md）完好无损
- ⚠️ 墨墨的 TOOLS.md 中可能有引用（可选清理，不清理也无害）

### Q2: 如果未来拆分，现在的知识会丢失吗？

**A:** 不会。知识分类存储，拆分时可以直接迁移：
```bash
# 拆分抖音 Agent 时
cp -r media/knowledge/平台规则/抖音规则库.md douyin/knowledge/
cp -r media/knowledge/案例分析/抖音爆款案例.md douyin/knowledge/
```

### Q3: 墨墨如何管理多个 Agent？

**A:** 墨墨通过文件队列协调：
- 分配任务：写入 `Agent/inbox/pending.md`
- 监控状态：读取 `Agent/status.md`
- 审查输出：读取 `Agent/outbox/completed.md`

### Q4: 多个 Agent 的 API 成本会增加吗？

**A:** 取决于使用方式：
- 如果共用 API Key：成本合并计算
- 如果独立 API Key：分别计算
- 墨墨可以帮哥哥监控和优化成本

### Q5: 如何保证知识不冲突？

**A:** 知识分类存储：
- 抖音的归抖音（`knowledge/平台规则/抖音/`）
- B 站的归 B 站（`knowledge/平台规则/B 站/`）
- 通用知识单独存放（`knowledge/通用/`）

### Q6: 扩展需要推倒重来吗？

**A:** 不需要。扩展是平滑演进的：
1. 创建新 Agent
2. 迁移相关知识
3. 迁移进行中的任务
4. 测试验证
5. 切换流量

### Q7: 墨墨会主动提醒扩展时机吗？

**A:** 会。墨墨会：
- 每天监控任务完成情况
- 每周生成运营报告
- 发现瓶颈时主动提醒
- 提供数据支持和扩展建议
