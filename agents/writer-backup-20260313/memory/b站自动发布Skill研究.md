## 🎯 核心设计理念

### 1. 战略性暂停（Strategic Pause）

> **真正的自动化不是消灭人的参与，而是让人在最关键的决策点介入**

**做法：**
- 检测到未登录时 → **停下来让用户手动登录**，而不是自动填充密码
- 最终发布前 → 让用户确认内容

**原因：**
- 账号安全（密码不交给脚本）
- 风控问题（自动登录容易被封）
- 验证码难以绕过
- 把控制权交还给最了解上下文的人

**对咱们的启发：**
- 小媒 Agent 发布内容前，让用户做最终确认
- 登录、授权等敏感操作交给用户
- AI 处理繁琐的重复劳动，人做关键决策

---

### 2. 低风险 vs 高风险决策

| 低风险（AI 直接做） | 高风险（需要人工确认） |
|-------------------|---------------------|
| 选分区/话题 | 删除数据 |
| 调格式 | 支付金额 |
| 生成标题建议 | 发布内容确认 |
| 标签推荐 | 账号登录 |
| 封面裁剪建议 | 定时发布时间 |

**判断标准：**
- 是否可逆？（可逆 → 低风险）
- 是否涉及安全？（涉及 → 高风险）
- 是否涉及金钱？（涉及 → 高风险）
- 出错成本？（低 → AI 做，高 → 人确认）

---

### 3. 降级路径设计（Fallback Strategy）

**主方案：** macOS 系统剪贴板
- 优雅简洁
- 利用操作系统原生能力
- 代码简洁，兼容性好

**备用方案：** CDP WebSocket 直接注入
- 复杂但可靠
- 需要分块传输（6KB 以下）
- 动态定位 WebSocket URL
- 构造 ClipboardEvent

**核心思想：**
> 真实世界的软件不会总是运行在理想环境中，提前设计降级路径

---

### 4. 共性抽象（Common Abstraction）

**发现：** B 站专栏和小站底层用**同一个富文本编辑器组件**

**做法：**
- 识别共性流程：粘贴 → 验证 → 发布
- 封装差异：导航逻辑、入口不同
- 新增平台时只需扩展导航逻辑

**对咱们的启发：**
- 小红书、B 站、抖音可能有相似的编辑器组件
- 抽象出通用的"发布流程"接口
- 每个平台只实现差异部分

## 🏗️ 架构设计

### 核心流程

```
用户指令 → 解析意图 → 检查登录 → [未登录则暂停] → 
导航到发布页 → 填写标题 → 粘贴内容 → 选择分区 → 
验证 → [用户确认] → 发布
```

### 模块化设计

```
├── 导航模块（平台差异）
│   ├── B 站专栏入口
│   └── B 站小站入口
├── 编辑器模块（共性）
│   ├── 标题填写
│   ├── 内容粘贴
│   └── 格式验证
├── 决策模块
│   ├── 分区选择（AI 自动）
│   ├── 标题生成（AI 建议）
│   └── 发布确认（用户）
└── 降级策略
    ├── 主方案：系统剪贴板
    └── 备用：CDP 注入
```

## 📋 技术细节

### CDP 注入方案

**问题：** WebSocket payload 限制 6KB

**解决：**
- 分块传输
- 动态定位 WebSocket URL
- 构造 ClipboardEvent

**代码结构：**
```javascript
// 备用方案：CDP 注入
async function pasteViaCDP(content) {
  // 1. 分块（每块 < 6KB）
  const chunks = splitContent(content, 6000);
  
  // 2. 获取编辑器 WebSocket URL
  const wsUrl = await getEditorWSUrl();
  
  // 3. 逐块注入
  for (const chunk of chunks) {
    await injectChunk(wsUrl, chunk);
  }
}
```

### 分区选择 AI 决策

**输入：** 标题 + 前 200 字正文

**输出：** 推荐分区

**策略：**
- 语义匹配
- 不询问用户，直接选择
- 允许事后修改（可逆）
