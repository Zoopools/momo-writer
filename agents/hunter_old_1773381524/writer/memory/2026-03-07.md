- **时间:** 18:00-19:02
- **问题:** 墨墨在飞书上无法回复消息，小媒没有聊天窗口
- **排查过程:**
  1. 检查配置 → 发现环境变量未展开 (`${OPENCLAW_FEISHU_*}` 是字符串)
  2. 修复配置 → 填入实际 App ID 和 App Secret
  3. 重启 Gateway → 仍然无法回复
  4. 深入排查 → 发现飞书官方插件需要 OAuth 授权
  5. 最终发现 → 配置格式应该是 `bots[]` 数组格式
- **根本原因:** 
  - App ID 配置使用环境变量引用但未被展开
  - 飞书官方插件 (v2026.3.7-beta.1) 需要多 bot 数组格式
- **解决方案:**
  ```json
  {
    "channels": {
      "feishu": {
        "enabled": true,
        "bots": [
          {
            "appId": "cli_a922eb10d3345bd9",
            "appSecret": "RmuIQUydfwh3s2sMYyaPCgG0a0PNWDvq",
            "agentId": "writer"
          },
          {
            "appId": "cli_a92173401c385bc0",
            "appSecret": "5fThCqKr3xBUwTwTUFe45cNvSvLHrRJt",
            "agentId": "media"
          }
        ]
      }
    }
  }
  ```
- **关键配置:**
  - 使用 `channels.feishu.bots[]` 数组格式
  - 每个 bot 配置 `appId`, `appSecret`, `agentId`
  - 飞书官方插件替代内置插件
- **状态:** ✅ 19:02 修复成功，墨墨能正常回复飞书消息

## 🔧 配置修复经验总结

### 问题诊断流程
1. **检查日志** → `tail -f /tmp/openclaw/openclaw-*.log`
2. **检查配置** → `cat ~/.openclaw/openclaw.json | python3 -m json.tool`
3. **检查插件** → `openclaw plugins list`
4. **检查通道** → `openclaw channels status`
5. **检查 Gateway** → `openclaw status`

### 常见陷阱
1. **环境变量未展开** → 配置中 `${VAR}` 不会被自动替换，需要填入实际值
2. **配置格式错误** → 2026.3.2 版本需要 `bots[]` 数组格式
3. **插件冲突** → 内置插件和官方插件不能同时启用
4. **OAuth 授权** → 飞书官方插件需要用户授权才能发送消息

### 关键命令
```bash
# 检查配置
cat ~/.openclaw/openclaw.json | python3 -m json.tool

# 检查插件
openclaw plugins list

# 检查通道
openclaw channels status

# 重启 Gateway
openclaw gateway restart
```

```json
        }
      }
    }
  },
  "bindings": [
    { "agentId": "writer", "match": { "channel": "feishu", "accountId": "writer" } },
    { "agentId": "media", "match": { "channel": "feishu", "accountId": "media" } }
  ]
}
```

## 🚨 深夜 Gateway 故障与修复 (22:00-23:00) ⭐⭐⭐⭐⭐

### WebUI 闪断问题排查
- **时间:** 22:00-23:00
- **现象:** WebUI 隔一会就无法访问，然后过几秒自动恢复
- **根本原因:** 
  1. **crontab 脚本与 LaunchAgent 冲突** - auto-fix.sh 每分钟执行，与 LaunchAgent 同时尝试重启 Gateway
  2. **进程争夺端口** - 两个进程争夺 18789 端口，导致其中一个被系统杀死
  3. **配置频繁重写** - 脚本频繁修改配置触发 watchConfig 保护机制
  4. **WebSocket 强制断开** - 1006 abnormal closure 错误
  5. **锁文件占用** - gateway.lock 文件被僵尸进程占用，导致 5000ms timeout

#### 故障诊断流程
```bash
# 1. 检查 Gateway 状态
openclaw status --deep

# 2. 查看日志
tail -f ~/.openclaw/logs/gateway.log | grep -i "error\|close\|abnormal"

# 3. 检查进程
ps aux | grep openclaw

# 4. 检查端口
lsof -i :18789
```

#### 修复步骤
```bash
# 1. 移除 crontab 中的 auto-fix 任务
(crontab -l | grep -v "auto-fix.sh") | crontab -

# 2. 清理残留进程
kill -9 $(pgrep -f openclaw) 2>/dev/null

# 3. 清理锁文件
rm -f ~/.openclaw/gateway.lock

# 4. 重装 LaunchAgent
openclaw gateway install --force

# 5. 重启 Gateway
openclaw gateway start --force --allow-unconfigured
```

#### 关键教训
1. **crontab 脚本与 LaunchAgent 会冲突** - 不要同时使用两种方式管理 Gateway
2. **自动修复脚本可能适得其反** - 过于频繁的修复会导致系统不稳定
3. **gateway.lock 文件会导致启动失败** - 崩溃后需要清理
4. **配置验证很重要** - 修改前先用 `openclaw config validate`
5. **LaunchAgent 已经提供崩溃自动重启** - 不需要额外的自动修复脚本

```bash
#!/bin/bash
# 墨墨专属 macOS 监控脚本（增强版：飞书 + 系统通知）
LOG_FILE=~/.openclaw/logs/gateway-monitor.log
BROTHER_OPEN_ID="ou_44a485e0921ccfa29b3fa386eb33eff7"

check_gateway() {
    if ! ps aux | grep -q "[o]penclaw-gateway"; then
        return 1
    fi
    if ! openclaw status 2>&1 | grep -q "Gateway.*reachable"; then
        return 1
    fi
    if ! openclaw status 2>&1 | grep -q "Feishu.*ON.*OK"; then
        return 1
    fi
    return 0
}

send_notify() {
    local msg="$1"
    osascript -e "display notification \"$msg\" with title \"🦞 OpenClaw 监控\""
    # 飞书通知（如果 Gateway 还能用）
    openclaw message send --target "$BROTHER_OPEN_ID" --channel feishu --message "$msg" 2>/dev/null
    echo "[$(date)] ❌ $msg" >> "$LOG_FILE"
}

main() {
    if check_gateway; then
        echo "[$(date)] ✅ Gateway 运行正常" >> "$LOG_FILE"
    else
        send_notify "Gateway 异常！进程/端口/状态不匹配，请检查"
    fi
}

main
```
