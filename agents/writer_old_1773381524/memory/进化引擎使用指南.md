# 进化引擎使用指南

## 🎯 快速决策树

```
经验/错误发生
    ↓
问自己 5 个问题：
    ↓
1. 这是代码 bug 吗？
   ├─ 是 → evolver
   └─ 否 → 继续
    ↓
2. 需要自动修复吗？
   ├─ 是 → evolver
   └─ 否 → 继续
    ↓
3. 这是配置/文档类经验吗？
   ├─ 是 → self-improving-agent
   └─ 否 → 继续
    ↓
4. 这个问题会重复出现吗？
   ├─ 是 → evolver (编码为 Gene)
   └─ 否 → 继续
    ↓
5. 需要跨项目复用吗？
   ├─ 是 → evolver (Genes 可共享)
   └─ 否 → self-improving-agent
```

## 📊 决策矩阵

### 使用 self-improving-agent 的场景 ✅

| 场景 | 示例 | 理由 |
|------|------|------|
| **配置经验** | 飞书多 Agent 配置格式 | 文档类知识，不需要自动修复 |
| **工作流程** | 任务分配流程（墨墨→小媒） | 流程记录，人工遵循即可 |
| **最佳实践** | 成本优化技巧 | 经验总结，不需要自动化 |
| **故障排查** | 常见问题排查清单 | 参考文档，人工使用 |
| **用户偏好** | 哥哥喜欢怎么称呼 | 个人信息，不需要进化 |
| **一次性问题** | 某次特定的配置错误 | 不会重复，记录即可 |

**特征判断：**
- ✅ 文档类知识
- ✅ 人工阅读为主
- ✅ 不需要自动执行
- ✅ 本地经验即可
- ✅ 安全优先（只读）

---

### 使用 evolver 的场景 🧬

| 场景 | 示例 | 理由 |
|------|------|------|
| **代码 bug** | 某个函数反复崩溃 | 需要自动生成补丁 |
| **重复错误** | 配置格式反复出错 | 编码为 Gene，自动修复 |
| **性能问题** | 某操作耗时过长 | 自动优化代码 |
| **模式识别** | 多次出现类似问题 | 识别模式，自动预防 |
| **跨项目复用** | 通用配置验证逻辑 | Genes 可共享 |
| **自动化需求** | 需要自动检测修复 | evolver 自动化能力强 |

**特征判断：**
- ✅ 代码级问题
- ✅ 会重复出现
- ✅ 需要自动修复
- ✅ 可编码为模式
- ✅ 可接受一定风险

## 🔍 墨墨的分析流程

当遇到问题/经验时，墨墨会这样分析：

### 步骤 1：问题分类

```bash
# 问自己：
1. 这是什么类型的问题？
   - 代码 bug？
   - 配置错误？
   - 工作流程？
   - 用户偏好？

2. 问题的根源是什么？
   - 代码逻辑错误？
   - 配置格式不对？
   - 缺少文档？
   - 人为疏忽？
```

### 步骤 2：影响评估

```bash
# 问自己：
1. 这个问题会重复出现吗？
   - 高频重复 → evolver
   - 偶尔出现 → self-improving-agent

2. 影响范围多大？
   - 单点问题 → self-improving-agent
   - 系统性问题 → evolver

3. 修复成本高吗？
   - 低成本（记录即可）→ self-improving-agent
   - 高成本（需要代码变更）→ evolver
```

### 步骤 3：自动化需求

```bash
# 问自己：
1. 需要自动检测和修复吗？
   - 需要 → evolver
   - 不需要 → self-improving-agent

2. 人工处理可行吗？
   - 可行 → self-improving-agent
   - 不可行（太频繁/太复杂）→ evolver
```

### 步骤 4：复用性评估

```bash
# 问自己：
1. 这个经验可以跨项目复用吗？
   - 可以 → evolver (Genes)
   - 本地即可 → self-improving-agent

2. 对其他人有价值吗？
   - 有 → evolver (可共享)
   - 个人经验 → self-improving-agent
```

## 📋 实战案例分析

### 案例 1：飞书多 Agent 配置 ✅

**问题：** 配置格式从 `bots[]` 改为 `accounts + bindings`

**墨墨的分析：**
```
1. 问题类型：配置经验（文档类）
2. 重复频率：低（配置一次即可）
3. 自动化需求：不需要自动修复
4. 复用性：中等（其他人也需要）

决策：self-improving-agent ✅
理由：文档类知识，记录到 LEARNINGS.md 足够
补充：创建独立 Skill 供他人使用
```

**实际操作：**
- ✅ 记录到 `.learnings/LEARNINGS.md`
- ✅ 创建 `feishu-multiagent-onboard` Skill
- ✅ 发布到 GitHub + ClawHub

---

### 案例 2：代码函数反复崩溃 🧬

**假设问题：** 某个图片处理函数反复崩溃

**墨墨的分析：**
```
1. 问题类型：代码 bug
2. 重复频率：高（每次都崩）
3. 自动化需求：需要自动修复
4. 复用性：高（其他项目也需要）

决策：evolver 🧬
理由：代码级问题，需要自动生成补丁
```

**evolver 会：**
1. 扫描日志识别崩溃模式
2. 选择合适的 Gene（如"错误处理增强"）
3. 生成补丁代码
4. 验证补丁（运行测试）
5. 应用补丁（或等待审核）

---

### 案例 3：配置验证逻辑 🧬

**问题：** 飞书配置验证逻辑复杂，反复出错

**墨墨的分析：**
```
1. 问题类型：配置 + 代码混合
2. 重复频率：中（每次新配置都可能错）
3. 自动化需求：需要自动验证
4. 复用性：高（其他人也需要验证）

决策：混合方案 🎯
- 配置经验 → self-improving-agent
- 验证逻辑 → evolver (编码为验证 Gene)
```

## 🎯 墨墨的默认策略

**日常使用：**
```
1. 遇到问题 → 先记录到 self-improving-agent
2. 定期回顾（每周）→ 识别可编码为 Gene 的模式
3. 重要模式 → 用 evolver 编码为 Genes
```

**紧急情况：**
```
1. 严重 bug → 直接用 evolver 修复
2. 配置错误 → 先手动修复，再记录经验
```
